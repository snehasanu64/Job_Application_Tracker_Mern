Job Application Tracker

Full-Stack MERN Project — Interview Reference Guide

MongoDB Express.js React 18 Node.js Vite Axios Mongoose

1. Project Overview

A full-stack web application that helps users track their job applications. Users can add job applications with company name, role, and date applied. They can update the status of each application (Applied → Interview → Offer → Rejected) and delete entries they no longer need.

Key Features

  • Add new job applications with company, role, date, and status
  • Update application status (Applied, Interview, Offer, Rejected)
  • Delete applications
  • Filter applications by status
  • Dashboard with real-time statistics
  • Auto-retry & timeout handling for reliable data saving

2. Architecture

[ React Frontend ]   — Axios HTTP Requests →   [ Express API ]   — Mongoose →   [ MongoDB Atlas ]
Vercel (Port 3000)                               Render (Port 5000)                           Cloud Database
How it works: The React frontend sends HTTP requests (GET, POST, PUT, DELETE) via Axios to the Express backend. The backend uses Mongoose to read/write data from a MongoDB Atlas cloud database, then sends JSON responses back to the frontend.

5. Backend — File by File

📄 backend/server.js — Entry Point

What it does: Initializes Express, sets up middleware, registers routes, and connects to MongoDB.

Key Concepts

  • cors() — Allows the React frontend (different port) to talk to this server (Cross-Origin Resource Sharing)
  • express.json() — Parses incoming JSON request bodies from the frontend
  • app.use("/api/jobs", jobRoutes) — Mounts all job-related routes under /api/jobs
  • Auto-reconnect: If MongoDB connection drops, it retries every 5 seconds instead of crashing
  • Server starts before DB: Express starts listening immediately so Render doesn't kill the process
// Key connection logic
const connectDB = async () => {
  try {
    await mongoose.connect(process.env.MONGO_URI, {
      serverSelectionTimeoutMS: 10000,
      socketTimeoutMS: 45000,
    });
    console.log("Connected to MongoDB");
  } catch (err) {
    console.log("Retrying connection in 5 seconds...");
    setTimeout(connectDB, 5000);  // Retry instead of crash
  }
};
📄 backend/models/Job.js — Mongoose Schema

What it does: Defines the structure (schema) of a Job document in MongoDB.

const jobSchema = new mongoose.Schema({
  company:     { type: String, required: true, trim: true },
  role:        { type: String, required: true, trim: true },
  dateApplied: { type: String, required: true },
  status:      { type: String, 
                 enum: ["Applied", "Interview", "Rejected", "Offer"],
                 default: "Applied" }
}, { timestamps: true });  // Adds createdAt & updatedAt

Key Concepts

  • required: true — Validation: company, role, and date are mandatory fields
  • enum — Restricts status to only 4 valid values
  • trim: true — Removes whitespace from the beginning and end of strings
  • timestamps: true — Mongoose automatically adds createdAt and updatedAt
📄 backend/routes/jobRoutes.js — API Routes (CRUD)

What it does: Handles all 4 REST API operations for job applications.

MethodEndpointMongoose MethodDescription
GET/api/jobsJob.find().sort()Fetch all jobs (newest first)
POST/api/jobsnew Job().save()Create a new job application
PUT/api/jobs/:idJob.findByIdAndUpdate()Update a job's status
DELETE/api/jobs/:idJob.findByIdAndDelete()Delete a job application

Reliability Feature — DB Check Middleware

// Checks if DB is connected before any operation
const checkDB = (req, res, next) => {
  if (mongoose.connection.readyState !== 1) {
    return res.status(503).json({ 
      message: "Database is connecting, please try again." 
    });
  }
  next();
};

This prevents the server from crashing if a user tries to save data while the database is still connecting.

6. Frontend — File by File

📄 frontend/src/api.js — API Client

What it does: Creates an Axios instance with 30-second timeout and automatic retry (up to 2 retries).

Why?

  • Timeout: Render's free tier takes 30-60s to wake up (cold start). Without a long timeout, requests fail.
  • Retry: If the backend returns a 503 (DB still connecting) or a network error, the frontend automatically waits 2 seconds and retries instead of showing an error.

7. Data Flow — How "Add Job" Works

  1. User fills the form in JobForm.jsx and clicks "+ Add Job"
  2. JobForm calls onJobAdded(formData) which is the handleAddJob function from App.jsx
  3. handleAddJob calls addJob(formData) from api.js
  4. api.js sends a POST request with the form data as JSON to http://backend-url/api/jobs
  5. Express receives the request in jobRoutes.js → validates the data → creates a new Mongoose document
  6. Mongoose saves the document to MongoDB Atlas and returns the saved document with a generated _id
  7. Express sends the saved document as a JSON response (status 201) back to the frontend
  8. App.jsx adds the returned job to the beginning of the jobs state array
  9. React re-renders all components — the new card appears in the list and stats update automatically

8. Database — MongoDB Atlas

MongoDB is a NoSQL document database. Data is stored as JSON-like documents (BSON) in collections.

Sample Document in the "jobs" Collection

{
  "_id": "66708b7d4f9c3f2d2a456abc",    // Auto-generated unique ID
  "company": "Google",
  "role": "Frontend Developer",
  "dateApplied": "2026-06-18",
  "status": "Applied",
  "createdAt": "2026-06-18T00:26:00.000Z",  // Auto by timestamps
  "updatedAt": "2026-06-18T00:26:00.000Z",  // Auto by timestamps
  "__v": 0                                    // Version key
}
Why MongoDB? It stores data in a flexible, JSON-like format which maps naturally to JavaScript objects — perfect for a JavaScript full-stack app. No need to define rigid table schemas like SQL.

10. Common Interview Questions & Answers

Q: What is MERN stack?

A: MERN stands for MongoDB, Express.js, React, and Node.js. It's a JavaScript-only tech stack — JavaScript runs on the frontend (React), backend (Node + Express), and the database stores JSON-like documents (MongoDB). This means one language across the entire application.

Q: What is Mongoose and why use it?

A: Mongoose is an ODM (Object Data Modeling) library for MongoDB. It provides schema validation, type checking, and convenient query methods like find(), save(), findByIdAndUpdate(). Without Mongoose, we'd write raw MongoDB queries which are harder to maintain.

Q: What is CORS and why do you need it?

A: CORS stands for Cross-Origin Resource Sharing. Browsers block requests from one origin (localhost:3000) to another (localhost:5000) by default for security. The cors() middleware tells the browser "it's okay, allow requests from the frontend."

Q: What is a REST API?

A: REST (Representational State Transfer) is a design pattern for APIs. Each URL represents a resource (/api/jobs), and HTTP methods define the action: GET (read), POST (create), PUT (update), DELETE (remove). My backend follows this pattern.

Q: What are React Hooks? Which ones do you use?

A: Hooks let you use React features in functional components. I use: useState — to manage local state (jobs array, loading status, form data). useEffect — to fetch data from the backend when the component first mounts (runs once on page load).

Q: How do you handle errors?

A: Three layers: (1) Frontend: try/catch blocks in every handler + toast notifications for user feedback + automatic retry with Axios interceptors. (2) Backend: try/catch in every route + a DB check middleware that returns 503 if the database isn't ready. (3) Connection: Auto-reconnect logic in server.js retries the MongoDB connection every 5 seconds.

Q: What is the difference between SQL and NoSQL?

A: SQL databases (MySQL, PostgreSQL) use fixed table schemas with rows and columns. NoSQL databases like MongoDB store flexible JSON-like documents in collections. I chose MongoDB because my data naturally fits a JSON structure, and it pairs well with JavaScript.

Q: Why Vite instead of Create React App?

A: Vite is much faster than CRA. It uses native ES modules for instant dev server startup and Hot Module Replacement (HMR). CRA uses Webpack which is slower. Vite also produces smaller production bundles.
